# CTalkDiversity 현황 진단 리포트

## 1. 현재 프로젝트 성격

CTalkDiversity는 **정적 HTML 기반의 커뮤니티/행사 웹사이트**다.

핵심 역할은 다음 세 가지로 보인다:
- 커뮤니티/조직 소개
- 지난 행사(Conference) 아카이브
- Call for Speakers 공지/참여 유도

기술적으로는 HTML5 UP 템플릿을 기반으로 구성된 **정적 사이트**이며, 현재는 HTML/CSS/이미지 자산을 직접 관리하는 방식이다.

---

## 2. 현재 잘 되어 있는 점

### 2.1 구조가 단순해서 이해하기 쉽다
- `index.html`이 메인 진입점
- `Pages/`에 상세 페이지 모음
- `Conferences/`, `CallForSpeakers/`, `images/`에 자산 정리
- `assets/`에 공통 스타일/스크립트 정리

초기 운영 단계에서는 이 단순성이 큰 장점이다.

### 2.2 이미 실제 콘텐츠가 들어가 있다
- About / Purpose / Mission / Vision
- 24th / 25th / 26th 행사 페이지
- Call for Speakers 페이지들

즉, 단순 템플릿 수준이 아니라 실제 운영 가능한 사이트 상태다.

### 2.3 정적 사이트라 배포가 쉽다
- 서버 의존성이 낮음
- GitHub Pages 같은 환경에 적합
- 백엔드 없이도 유지 가능

### 2.4 콘텐츠 성격이 명확하다
사이트가 무엇을 하려는지 비교적 분명하다.
- Cultural Talk For Diversity 소개
- 과거 활동 기록
- 미래 참여 유도

이건 앞으로 사이트를 키워갈 때 중요한 자산이다.

---

## 3. 현재 보이는 문제점 / 리스크

### 3.1 콘텐츠가 늘어날수록 수작업 부담이 커진다
현재는 행사 하나를 추가할 때마다:
- 메인 페이지 수정
- conferences 목록 페이지 수정
- 개별 상세 페이지 생성
- 이미지 배치
- 링크 점검

이런 작업을 모두 직접 해야 한다.

행사가 계속 늘어나면 반복 작업이 많아지고 실수가 늘어날 가능성이 높다.

### 3.2 정보가 여러 파일에 분산되어 있다
예를 들어 conference 하나가 추가될 때 관련 정보가 여러 파일에 퍼질 수 있다.
- index.html
- Pages/conferences.html
- Pages/Conference/conference-xx.html
- Conferences/xx/* 이미지

즉, 콘텐츠 1건당 관리 포인트가 많다.

### 3.3 네이밍/자산 관리 일관성이 약해질 조짐이 있다
예시:
- `Conferences/26/24th-01-main.jpeg`

회차와 파일명이 직관적으로 일치하지 않는 흔적이 있다.
이런 패턴은 시간이 지날수록 관리 혼란을 만든다.

### 3.4 템플릿 기반 구조가 반복 수정에 취약하다
정적 HTML 직접 수정 방식은 초반엔 빠르지만,
다음 같은 요구가 오면 점점 비효율적이 된다.
- 카드 레이아웃 일괄 변경
- CTA 문구 일괄 수정
- 공통 메타 태그 추가
- 소셜 공유 최적화
- SEO 기본 정리

### 3.5 README가 실제 프로젝트 설명 역할을 못 한다
현재 `README.txt`는 HTML5 UP 원본 템플릿 설명이다.
즉, 이 repo 자체가 무엇인지, 어떻게 관리하는지, 어떤 정책으로 콘텐츠를 추가하는지 설명하는 문서가 부족하다.

## 4. 현재 단계에서의 핵심 과제

이 프로젝트의 핵심 과제는 단순히 "페이지를 몇 개 더 만든다"가 아니라,
**운영 가능한 정적 사이트를 점점 더 효율적으로 관리 가능한 구조로 발전시키는 것**이다.

즉, 과제는 크게 두 축이다.

### 4.1 콘텐츠 운영성 향상
- 새 행사 추가를 쉽게 만들기
- 콜포스피커 공지 갱신 쉽게 만들기
- 링크/자산/문구 관리 실수 줄이기

### 4.2 구조적 유지보수성 향상
- 파일 구조 정리
- 문서화
- 반복 작업 표준화
- 필요 시 정적 사이트 생성기 도입 검토

---

## 5. 우선순위 높은 개선 5개

## Priority 1 — 프로젝트 README를 실제 문서로 교체/보강
가장 먼저 해야 할 일이다.

필요 내용:
- 이 사이트가 무엇인지
- 폴더 구조 설명
- 새 conference 추가 방법
- 새 call for speakers 추가 방법
- 이미지 파일 어디에 두는지
- 배포 방식이 무엇인지

효과:
- 미래의 작업 효율 증가
- 협업 가능성 증가
- 프로젝트 방향성 명확화

---

## Priority 2 — 콘텐츠 추가 규칙 표준화
conference / call-for-speakers를 추가하는 규칙을 문서화해야 한다.

예:
- 파일명 규칙
- 이미지명 규칙
- 회차 폴더 규칙
- index / listing / detail 페이지 수정 순서
- 링크 점검 체크리스트

효과:
- 수작업이더라도 덜 헷갈림
- 누락/오타/경로 실수 감소

---

## Priority 3 — 자산 네이밍/폴더 정리
이미지 및 파일명 체계를 정리해야 한다.

예:
- `Conferences/26/26th-01-main.jpeg`
- `Conferences/26/26th-02-speaker-name.jpeg`

같은 식으로 회차와 파일명이 일치하도록 규칙화하면 좋다.

효과:
- 유지보수 피로 감소
- 파일 찾기 쉬움
- 장기 운영 안정성 증가

---

## Priority 4 — 공통 UI/콘텐츠 구조 점검
현재 페이지들이 잘 열리긴 하지만, 공통 요소를 다시 점검할 필요가 있다.

체크 포인트:
- navigation 일관성
- 버튼/문구 통일감
- home / conferences / call-for-speakers 사이 흐름
- mobile에서 읽기 쉬운지
- 이미지/제목/요약 길이 균형

효과:
- 사용자 경험 개선
- 사이트 완성도 향상

---

## Priority 5 — 장기적으로 데이터 중심 구조로 갈지 검토
지금 당장 프레임워크로 갈 필요는 없어 보인다.
하지만 행사가 계속 늘어나면 아래를 검토해야 한다.
- Jekyll
- Eleventy
- Astro
- 혹은 JSON/YAML 기반 콘텐츠 분리

이 단계는 지금 즉시보다 **운영 규칙이 먼저 안정된 후** 검토하는 게 좋다.

효과:
- 반복 작업 자동화 가능
- 템플릿 재사용 쉬움
- 장기 유지보수성 개선

---

## 6. 추천 진행 순서

### Step 1
프로젝트 README를 실제 운영 문서로 만든다.

### Step 2
conference / call-for-speakers 추가 규칙 문서를 만든다.

### Step 3
현재 파일/이미지 네이밍 정리를 위한 체크리스트를 만든다.

### Step 4
사이트 구조/UI 점검 후 소규모 정리 작업을 한다.

### Step 5
사이트가 더 커질 조짐이 보이면 정적 사이트 생성기 전환 여부를 검토한다.

---

## 7. 지금 시점의 결론

CTalkDiversity는 이미 콘텐츠를 담을 수 있는 좋은 출발점 위에 있다.
문제는 "사이트가 없다"가 아니라,
**앞으로 계속 커질 때 지금 방식으로도 버틸 수 있도록 운영 체계를 만들어야 한다**는 쪽이다.

즉, 지금 해야 할 일은 화려한 리빌드보다 먼저:
- 구조 파악
- 운영 규칙 문서화
- 반복 작업 표준화
- 자산 정리 기준 확립

이렇게 기초 체계를 만드는 것이다.
